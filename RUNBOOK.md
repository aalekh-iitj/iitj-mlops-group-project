## STEP 1 — Local environment (Python 3.14 problem solved)

The doc pins **Python 3.11** in the CI workflow, and `torch` / `transformers` / `wandb` wheels do **not yet support Python 3.14** as of June 2026. So:

### 1A. **Create a Python 3.11 venv for local sanity-checking ONLY.**
You only need this on your laptop to:
- run `prepare_dataset.py` (Task 2)
- build & test the Docker image locally (Task 6)
- lint the code

Training and inference themselves run on Kaggle / in Docker / in GitHub Actions, where the Python version is controlled for you.

Pick whichever you have:

```bash
# Option A — pyenv (recommended on macOS/Linux)
pyenv install 3.11.9
pyenv local 3.11.9
python -m venv .venv
source .venv/bin/activate
python --version     # should print Python 3.11.9
pip install -r requirements.txt
```

```bash
# Option B — conda (works on Windows too)
conda create -n mlops-a3 python=3.11 -y
conda activate mlops-a3
pip install -r requirements.txt
```

```bash
# Option C — Windows, no pyenv
# Download Python 3.11.x from python.org, then:
py -3.11 -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

### 1B. **What if a teammate insists on Python 3.14?**
They will hit `ERROR: Could not find a version that satisfies the requirement torch` from pip. Tell them to use the 3.11 venv above for any local work. **All heavy lifting happens in Kaggle/Docker/CI, so the laptop Python version is mostly cosmetic.**

### 1C. **Why "local" is needed at all**
| Task | Where it runs | Why |
|---|---|---|
| Data prep (Task 2) | Local **or** Kaggle CPU notebook | One-time, fast, reproducible |
| Training (Task 4) | **Kaggle GPU only** | Free T4, 30h/week; CI is too slow + no GPU |
| Model push to HF (Task 5) | Inside training script on Kaggle | Uses HF_TOKEN from Kaggle Secrets |
| Docker build/test (Task 6) | Local | Smoke test before pushing |

## STEP 3 — Data preparation (Task 2, 15 marks)

We're using **AG News** (`fancyzhx/ag_news` on HF Hub) — a 4-class topic classification dataset. It fits the "text classification, well-scoped" example, stays under 50k samples, and is small enough to train on a free Kaggle T4 in minutes.

```bash
# Inside the 3.11 venv from STEP 1
python prepare_dataset.py
```

This writes:
- `data/processed/train.csv` (text, label columns)
- `data/processed/test.csv`
- `data/id2label.json` ← **committed** (mapping file)
- `data/label2id.json` ← **committed**
- `data/data_summary.json` ← **committed** (for the report's "Data cleaning decisions" section)

Commit only the three JSONs and the script. The CSVs are in `.gitignore` (large-ish, regenerated).

### What the cleaning script does (paste into the report, Task 2)
- Lowercase is left to the tokenizer (DistilBERT does its own cased/uncased normalisation).
- Stripped whitespace, collapsed internal whitespace, dropped empties and exact-duplicate texts.
- Stable integer label encoding based on HF's original sort order — keeps `id2label.json` deterministic.
- Took 40,000 rows from the train split (well under 50k as required) and did an 80/20 train/test split with `random_state=42` for reproducibility.

`distilbert-base-uncased` (≈67M params, ≈250 MB) — cite the HF model card in the report.

### 7.4 — Why the Dockerfile looks the way it does (paste into the report, Task 6 design)
- `python:3.11-slim` — small base, exact version pinned so wheels resolve.
- Installs only inference deps (torch, transformers, tokenizers, hf-hub) instead of the full `requirements.txt` — drops wandb, datasets, flake8 → smaller image, faster pull.
- `ARG HF_MODEL_NAME` with default `distilbert-base-uncased` — image is buildable with no model, swap in your real model at build time.
- `ENV HF_MODEL_NAME=…` so the runtime can read it (also over-rideable at `docker run -e HF_MODEL_NAME=…`).
- Pre-downloads model weights at build time so the image is self-contained; falls back to runtime download if the build-time pull fails (e.g. private repo without a build secret).
- Runs as `python -m src.inference` (package mode) so relative imports work.
